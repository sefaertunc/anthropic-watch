# v1.4.0 Live Smoke Test — Production Validation Checklist

**Purpose:** Validate v1.4.0 against real production data over three consecutive clean scheduled runs, per the v1.4.0 prompt's Post-completion verification step 8. No code changes — this is observation only.

**Pass criteria:** Three consecutive runs (or manual triggers) with all twenty new sources either producing items or emitting `0` items via the documented graceful-skip path, zero `consecutiveFailures` ticks on any new source, clean dashboard render, clean feed envelopes.

**Fail criteria:** Any new source with `consecutiveFailures >= 1` across two consecutive runs, malformed feed output, dashboard break, or schema contract violations.

## Final sign-off

After three clean runs:

- [ ] v1.4.0 validated in production
- [ ] Client `1.0.1` validated (at least one community-category item returned through the client library — spot-test with a local script or proceed to Option B)
- [ ] No patch release needed

If any run fails, capture:

1. The failing source key
2. The error message from `run-report.json`
3. The workflow run URL
4. A hypothesis (Reddit UA? twitterapi.io outage? repo renamed? bot-filter misfire?)

Then decide: patch release (v1.4.1), revert (unlikely — community category is additive and safe to keep even if one source is flaky), or wait-and-observe (often correct for transient external API issues).

## Notes for future smoke tests

The backfill burst on Run 1 is expected and one-time. Runs 2 and 3 should show normal steady-state item volume. If item counts on Run 2 are unexpectedly high (comparable to Run 1), investigate: it would suggest dedup is failing somewhere.

The pattern here is general — adapt this checklist template for future `source-addition` releases (v1.5.0+). Keep it in `docs/` as `SMOKE-TEST-TEMPLATE.md` if you want to standardize.
