# v1.3.0 — Client Library (`@sefaertunc/anthropic-watch-client`)

> **Revision status:** post-review (Round 1, 2026-04-23). All blocking and material findings from the plan-reviewer pass have been incorporated. See the "Plan review record" section near the end of this document for a finding-by-finding accounting of changes from the pre-review draft. Implementation may proceed against this revision.

A significant release that extends anthropic-watch from a feed publisher into a feed publisher plus a first-party consumer SDK. This version introduces a new npm-published package at `packages/client/` within the monorepo, shipped as `@sefaertunc/anthropic-watch-client@1.0.0`. The scraper itself is unchanged — no new sources, no scraper logic changes, no schema changes beyond what v1.2.0 already delivered.

## Why we are doing this (context for plan review)

### The problem v1.3.0 solves

The v1.2.0 integration audit of Worclaude's consumption of anthropic-watch revealed a production bug: Worclaude deduplicated feed items on `id` alone rather than on the documented `${id}|${source}` composite key. The result was cross-source id collisions — the fixture shipped in v1.2.0 at `docs/fixtures/all.sample.json` contains a concrete example: `id: "2.1.117"` appears from both `claude-code-changelog` and `npm-claude-code`, which Worclaude's state machine would collapse into one.

v1.2.0 addressed the **documentation and data shape** side of this bug. The feed now pre-computes `uniqueKey: "${id}|${source}"` on every item. The schema documentation now shows a correct consumption pattern. Reference fixtures now exist for contract testing.

What v1.2.0 **did not** solve is the fact that every consumer of the feed must still hand-write the same fetch, version-gate, dedup, and state-persistence code. Worclaude did it once and got it wrong. Any future consumer will face the same blank page, with the same opportunity for the same bug class.

v1.3.0 solves this by shipping a small, zero-dependency client library that encapsulates the correct consumption pattern once, for everyone.

### Why an npm package (and the history of this decision)

Two earlier decisions in this project established that `anthropic-watch` itself — the scraper — should not be on npm. The scraper is infrastructure: it runs on GitHub Actions on a cron schedule, writes state into a git repo, and publishes feeds to GitHub Pages. You cannot meaningfully `npm install` infrastructure.

The **client library** is a different product. It is a small set of reusable functions and a client class that consumers import into their own code. That is exactly what npm is for.

This is a two-package repo going forward:

- **`anthropic-watch`** (the scraper, this repo root) — stays GitHub-only, never published to npm. Unchanged by this release except for monorepo restructuring.
- **`@sefaertunc/anthropic-watch-client`** (new, `packages/client/`) — published to npm at version 1.0.0 as part of this release. Separate versioning from the scraper going forward — scraper v1.3.0 ships alongside client v1.0.0; they are not locked together after that.

The package name is scoped under `@sefaertunc` deliberately. It avoids npm name squatting concerns, makes ownership explicit, and matches the pattern Anthropic themselves use (`@anthropic-ai/claude-code`).

### Why a monorepo rather than a second repo

A separate repository for the client would introduce cross-repo version drift, two CHANGELOGs to keep in sync during coordinated changes, and a harder path for running contract tests against the producer's reference fixtures (which live in this repo). A monorepo keeps the producer and the canonical consumer in one commit boundary, so schema changes and their library-side accommodations can land atomically.

The monorepo boundary is a convention, not a build tool decision. This release does **not** introduce Turborepo, Nx, Lerna, pnpm workspaces, or any other tool. Each package has its own `package.json`; the root `package.json` continues to own the scraper's dependencies and scripts exactly as it does today. Cross-package coordination is manual and deliberate. If and when the monorepo grows a third package, we can revisit tooling; two packages do not justify the weight.

### Why 1.0.0 for the client rather than 0.x

The client library's public API is narrow enough and well-motivated enough by the audit that we should commit to semver from day one. Anyone who adopts it needs to know that `1.0.0` → `1.x` is a safe upgrade path. A 0.x version signals instability that isn't present — the API is already shaped by the production need.

### Why this release doesn't migrate Worclaude

Worclaude's adoption of this library is a **Worclaude-side release** (planned as Worclaude v2.6.0). That work includes state schema migration from v2 to v3, template updates, and removing Worclaude's hand-rolled precheck script. It is not a blocker for this release. Shipping the library is complete without any downstream consumer; adoption is a separate commitment tracked separately.

You, Sefa, are working on Worclaude in parallel, so you have a reference consumer visible while the library is being designed. This is the correct ordering when the two are being developed concurrently — the library is shaped by real consumption while it is being built.

### What is explicitly out of scope for v1.3.0

- **Worclaude consumer migration.** Separate session, separate repo, separate PR cycle.
- **New scraper sources.** Deferred to v1.4.0 (planned: `github-commits` scraper for `anthropics/claude-cookbooks`, `anthropics/skills`, `anthropics/claude-plugins-official`; `reddit-subreddit` scraper for community subs).
- **RSS `guid` composite key change.** Explicitly deferred to v2.0 per v1.2.0's decision.
- **Feed envelope `version` bump.** Stays at `"1.0"` — client ships schema support for `"1.0"` only and throws on anything else.
- **TypeScript sources.** The library is plain ESM JavaScript with JSDoc type annotations that generate a `.d.ts` file at publish time. Consistent with the scraper's "no build step" philosophy.
- **Dependencies.** The library must have zero runtime dependencies. `fetch` is available natively in Node 18+ and all modern browsers. Any utility function the library needs must be written inline or deferred until a later release.
- **Browser vs Node-only.** Target Node 18+ for this release. Browser support is a follow-up concern — the library should not _prevent_ browser use, but we won't actively test it in v1.3.0.

### Public API of the client library

This is the entire public API v1.0.0 commits to. Nothing more, nothing less. Anything internal to the library lives in non-exported modules.

```js
// Named exports from @sefaertunc/anthropic-watch-client

// Classes
export class AnthropicWatchClient { … }

// Pure helpers
export function uniqueKey(item)           // (Item) => string
export function filterNew(items, seenSet) // (Item[], Set<string>) => Item[]
export function dedupe(items)             // (Item[]) => Item[]

// Typed errors
export class AnthropicWatchError          // base class for all library errors
export class FeedVersionMismatchError     // thrown when feed.version !== '1.0'
export class FeedFetchError               // thrown on network / HTTP failure
export class FeedMalformedError           // thrown when shape is unexpected

// Constants
export const SUPPORTED_FEED_VERSION = '1.0'
export const DEFAULT_BASE_URL = 'https://sefaertunc.github.io/anthropic-watch/'
```

#### `AnthropicWatchClient` class

```js
const client = new AnthropicWatchClient({ baseUrl?, fetch?, timeout? });
//   baseUrl: override the default feed host (useful for self-hosters)
//   fetch:   custom fetch implementation (for testing or non-Node environments)
//   timeout: request timeout in ms, default 10000

await client.fetchAllItems({ signal? });       // => Item[] from all.json, validated
await client.fetchSourceItems(sourceKey, { signal? }); // => Item[] from {source}.json
await client.fetchRunReport({ signal? });      // => RunReport, validated
client.filterNew(items, seenSet);              // instance-method mirror of the helper
```

Every fetch method version-gates the response, throws `FeedVersionMismatchError` on mismatch, throws `FeedMalformedError` on unexpected shape, and throws `FeedFetchError` on network/HTTP failure. All three inherit from `AnthropicWatchError` so consumers can catch one class to handle all library errors.

#### Pure helpers

`uniqueKey(item)` returns `item.uniqueKey ?? \`${item.id}|${item.source}\``. The fallback is critical: items from pre-v1.2.0 feeds (or from a consumer's own archived state) may not have `uniqueKey`. The helper always produces the correct value.

`filterNew(items, seenSet)` returns only items whose `uniqueKey` is not in the provided Set. This is the canonical dedup operation.

`dedupe(items)` removes duplicates within a single array, keeping the first occurrence of each `uniqueKey`. Useful when merging feeds from multiple sources.

#### Typed errors

Consumers should be able to handle specific error types. For example:

```js
try {
  const items = await client.fetchAllItems();
} catch (err) {
  if (err instanceof FeedVersionMismatchError) {
    console.error(
      `Schema version changed to ${err.actualVersion}; update the library.`,
    );
  } else if (err instanceof FeedFetchError) {
    console.error(`Feed unavailable: ${err.cause?.message ?? err.message}`);
    // retry later
  } else if (err instanceof FeedMalformedError) {
    console.error(`Feed shape unexpected: ${err.message}`);
  } else {
    throw err; // unknown error, let it propagate
  }
}
```

Error classes carry structured context:

- `FeedVersionMismatchError` has `expectedVersion` (always `"1.0"` in v1.0.0) and `actualVersion`.
- `FeedFetchError` has `url`, `status` (nullable, for non-HTTP errors), and `cause` (the underlying error).
- `FeedMalformedError` has `url` and `reason` (human-readable shape description).

#### Task 1.6: Root `vitest.config.js` — exclude the client subpackage from root test runs

The repository currently has no `vitest.config.*` file; root `npm test` relies on Vitest defaults. Vitest's default `include` glob is `**/*.{test,spec}.?(c|m)[jt]s?(x)` and its default `exclude` list does **not** cover `packages/**`. Without a config, adding `packages/client/test/*.test.js` means root `npm test` would try to collect those tests without `npm ci` ever having been run inside `packages/client/` — a silent cross-package coupling that would break the moment the subpackage grew a unique devDep.

Create `vitest.config.js` at the repo root:

```js
import { defineConfig } from "vitest/config";

export default defineConfig({
  test: {
    exclude: ["**/node_modules/**", "**/dist/**", "packages/**"],
  },
});
```

This keeps the scraper's test command (`npm test` at root) behaving exactly as it does today — seventeen-ish files under `test/`, no client coupling. The client runs its own tests via `cd packages/client && npm test`. The two test suites are fully decoupled at the config level.

**Why not unify them under one root command?** It's tempting to have `npm test` run both suites. But that requires the root to have installed the subpackage's devDeps, which requires either npm workspaces (explicitly ruled out in this release) or a manual `npm ci` step in CI. Keeping them separate matches the monorepo philosophy of this release: two packages, manual coordination.

#### Task 2.2: `src/errors.js` — typed error classes

```js
export class AnthropicWatchError extends Error {
  constructor(message, options = {}) {
    super(message);
    this.name = "AnthropicWatchError";
    if (options.cause) this.cause = options.cause;
  }
}

export class FeedVersionMismatchError extends AnthropicWatchError {
  constructor(actualVersion, expectedVersion = "1.0") {
    super(
      `Feed version mismatch: expected "${expectedVersion}", got "${actualVersion}"`,
    );
    this.name = "FeedVersionMismatchError";
    this.actualVersion = actualVersion;
    this.expectedVersion = expectedVersion;
  }
}

export class FeedFetchError extends AnthropicWatchError {
  constructor(message, { url, status, cause } = {}) {
    super(message, { cause });
    this.name = "FeedFetchError";
    this.url = url;
    this.status = status ?? null;
  }
}

export class FeedMalformedError extends AnthropicWatchError {
  constructor(message, { url, reason } = {}) {
    super(message);
    this.name = "FeedMalformedError";
    this.url = url;
    this.reason = reason;
  }
}
```

Important implementation notes:

- `AnthropicWatchError` passes `cause` through to the underlying `Error` constructor via `super` options where supported (Node 16.9+). This is how ES2022 error causes chain. Tests must verify cause chaining works.
- Every subclass sets `this.name` explicitly. Without this, `err.name` is inherited from `Error` and stack traces become harder to read.
- `FeedFetchError.status` is `null` for non-HTTP errors (timeouts, DNS failures, etc.). Consumers can distinguish by checking `status === null`.

#### Task 2.4: `src/client.js` — `AnthropicWatchClient` and helpers

This is the meat of the library. Implement in this order to make test-driven work flow naturally:

**Step 1 — pure helpers**

```js
/**
 * @param {import('./types.js').Item} item
 * @returns {string}
 */
export function uniqueKey(item) {
  return item.uniqueKey ?? `${item.id}|${item.source}`;
}

/**
 * @param {import('./types.js').Item[]} items
 * @param {Set<string>} seenSet
 * @returns {import('./types.js').Item[]}
 */
export function filterNew(items, seenSet) {
  if (!(seenSet instanceof Set)) {
    throw new TypeError("filterNew: seenSet must be a Set");
  }
  return items.filter((item) => !seenSet.has(uniqueKey(item)));
}

/**
 * @param {import('./types.js').Item[]} items
 * @returns {import('./types.js').Item[]}
 */
export function dedupe(items) {
  const seen = new Set();
  const out = [];
  for (const item of items) {
    const key = uniqueKey(item);
    if (seen.has(key)) continue;
    seen.add(key);
    out.push(item);
  }
  return out;
}
```

The TypeError in `filterNew` is intentional. Passing an array instead of a Set is a common mistake — fail loudly.

**Step 2 — client class**

```js
import {
  FeedVersionMismatchError,
  FeedFetchError,
  FeedMalformedError,
} from "./errors.js";

// These are the source of truth; `src/index.js` re-exports them. Declaring
// them here (rather than in both places) prevents drift. Review finding N4.
export const SUPPORTED_FEED_VERSION = "1.0";
export const DEFAULT_BASE_URL = "https://sefaertunc.github.io/anthropic-watch/";

export class AnthropicWatchClient {
  /**
   * @param {Object} [options]
   * @param {string} [options.baseUrl]
   * @param {typeof fetch} [options.fetch]
   * @param {number} [options.timeout] - milliseconds
   */
  constructor(options = {}) {
    this.baseUrl =
      (options.baseUrl ?? DEFAULT_BASE_URL).replace(/\/$/, "") + "/";
    this.fetch = options.fetch ?? globalThis.fetch;
    this.timeout = options.timeout ?? 10000;

    if (typeof this.fetch !== "function") {
      throw new TypeError(
        "AnthropicWatchClient: no fetch implementation available. " +
          "Use Node 18+ or pass a custom fetch via options.fetch.",
      );
    }
  }

  async fetchAllItems({ signal } = {}) {
    const url = `${this.baseUrl}feeds/all.json`;
    const feed = await this.#fetchJson(url, { signal });
    this.#assertFeedShape(feed, url);
    return feed.items;
  }

  async fetchSourceItems(sourceKey, { signal } = {}) {
    if (typeof sourceKey !== "string" || sourceKey.length === 0) {
      throw new TypeError(
        "fetchSourceItems: sourceKey must be a non-empty string",
      );
    }
    const url = `${this.baseUrl}feeds/${encodeURIComponent(sourceKey)}.json`;
    const feed = await this.#fetchJson(url, { signal });
    this.#assertFeedShape(feed, url);
    return feed.items;
  }

  async fetchRunReport({ signal } = {}) {
    const url = `${this.baseUrl}feeds/run-report.json`;
    const report = await this.#fetchJson(url, { signal });
    this.#assertRunReportShape(report, url);
    return report;
  }

  filterNew(items, seenSet) {
    return filterNew(items, seenSet);
  }

  async #fetchJson(url, { signal } = {}) {
    const timeoutController = new AbortController();
    const timeoutId = setTimeout(() => timeoutController.abort(), this.timeout);
    const combinedSignal = signal
      ? anySignal([signal, timeoutController.signal])
      : timeoutController.signal;

    let response;
    try {
      response = await this.fetch(url, { signal: combinedSignal });
    } catch (cause) {
      clearTimeout(timeoutId);
      throw new FeedFetchError(`Network error fetching ${url}`, { url, cause });
    }
    clearTimeout(timeoutId);

    if (!response.ok) {
      throw new FeedFetchError(`HTTP ${response.status} fetching ${url}`, {
        url,
        status: response.status,
      });
    }

    try {
      return await response.json();
    } catch (cause) {
      throw new FeedMalformedError(`Response is not valid JSON`, {
        url,
        reason: cause.message,
      });
    }
  }

  #assertFeedShape(feed, url) {
    if (typeof feed !== "object" || feed === null) {
      throw new FeedMalformedError("Feed is not an object", {
        url,
        reason: `got ${typeof feed}`,
      });
    }
    if (feed.version !== SUPPORTED_FEED_VERSION) {
      throw new FeedVersionMismatchError(feed.version);
    }
    if (!Array.isArray(feed.items)) {
      throw new FeedMalformedError("Feed is missing items array", {
        url,
        reason: "feed.items is not an array",
      });
    }
  }

  #assertRunReportShape(report, url) {
    if (typeof report !== "object" || report === null) {
      throw new FeedMalformedError("Run report is not an object", {
        url,
        reason: `got ${typeof report}`,
      });
    }
    if (report.version !== SUPPORTED_FEED_VERSION) {
      throw new FeedVersionMismatchError(report.version);
    }
    if (!Array.isArray(report.sources)) {
      throw new FeedMalformedError("Run report is missing sources array", {
        url,
        reason: "report.sources is not an array",
      });
    }
    if (typeof report.summary !== "object" || report.summary === null) {
      throw new FeedMalformedError("Run report is missing summary", {
        url,
        reason: "report.summary is not an object",
      });
    }
  }
}

/**
 * Combines multiple AbortSignals into one that aborts when any of them do.
 *
 * Caveat — listener accumulation on long-lived signals:
 *   The `abort` listeners this function attaches to the input signals are
 *   released when the input signals themselves are garbage-collected. For
 *   short-lived user signals (or an absent user signal, the common case),
 *   this is a non-issue. For a pathological consumer that holds a single
 *   long-lived AbortSignal and fires thousands of concurrent fetches with
 *   it, listeners accumulate on that signal until it is either aborted or
 *   garbage-collected. Daily-cron consumers do not hit this shape. If a
 *   consumer reports it, promote anySignal to its own module and add a
 *   `removeEventListener` cleanup path. Review finding M4.
 */
function anySignal(signals) {
  const controller = new AbortController();
  for (const signal of signals) {
    if (signal.aborted) {
      controller.abort(signal.reason);
      return controller.signal;
    }
    signal.addEventListener("abort", () => controller.abort(signal.reason), {
      once: true,
    });
  }
  return controller.signal;
}
```

Implementation notes requiring judgment:

- **Timeout via AbortController.** Node 20+ has `AbortSignal.timeout(ms)` but we target 18+ so we compose manually. The `anySignal` helper is a small utility — placed in this file since it's the only consumer. If other places need it later, promote it.
- **`#private` methods.** ES private fields are Node 14.6+; fully supported at our target.
- **`encodeURIComponent(sourceKey)`.** Source keys come from trusted config but we still encode — a user-supplied value should not be trusted to be URL-safe.
- **Why `.replace(/\/$/, '') + '/'` on the baseUrl.** Normalizes whether or not the user passed a trailing slash. Prevents `//feeds/all.json` on some inputs.
- **Why throw `TypeError` from the constructor rather than a library error.** Constructor errors are programmer errors — wrong environment, wrong arguments. Library errors are runtime/network conditions. `TypeError` is the Node convention for programmer error.

#### Task 3.5: `test/fixtures.test.js` — fixture identity against producer references

This test guards against silent drift between the client's copy of the reference fixtures and the producer's canonical fixtures in `docs/fixtures/`.   it("all.valid.json equals docs/fixtures/all.sample.json", () => {

#### Task 3.6: `test/docs-example.test.js` — drift-protection for FEED-SCHEMA's Programmatic Consumption example

The project has a long-standing principle (v1.2.0 precedent) that the feed contract should be understandable and verifiable without runtime dependencies. `docs/FEED-SCHEMA.md`'s "Programmatic Consumption" section contains the canonical hand-rolled implementation of the consumption contract — non-JS consumers, air-gapped environments, and pre-adoption evaluators all depend on it being correct.

Task 5.1 (below) prepends a pointer note to that section recommending the library, but preserves the inline example as the canonical reference. This test enforces that the example stays correct _by CI_, rather than by discipline.

Mechanism:

1. Read `docs/FEED-SCHEMA.md` as text.
2. Locate the "Programmatic Consumption" section heading. Extract the first fenced JavaScript code block (\`\`\`js ... \`\`\`) that follows it.
3. Install runtime dependency seams: mock `fetch` to return `packages/client/fixtures/all.valid.json` content; replace any file-system state persistence (`STATE_PATH` reads/writes) with an in-memory `Map`.
4. Execute the extracted code by writing it to a temp file and `await import()`-ing it, OR by constructing an `AsyncFunction` from the source with the mocked globals bound into its scope. Prefer the temp-file + dynamic-import approach — it runs the example in a real ESM module context, which catches a wider class of bugs.
5. Assert:
   - No exception thrown during execution.
   - The `fresh` array (the variable the example produces) is non-empty.
   - Every item in `fresh` has a `uniqueKey` property that is a non-empty string.
   - Across two simulated runs (second run uses the seen-set produced by the first run), the seen-set grows monotonically and `fresh` shrinks to empty on the second run (since no new items have arrived).

**Two-run invariant: do it in a single module evaluation.** The naive approach is to `await import()` the wrapped example twice with cache-busting query strings and rely on Node re-executing the module. Don't. Node's ESM cache behavior for `file://` URLs with query strings is undocumented and has shifted across Node 20/22; more importantly, two fresh evaluations each create a fresh in-memory state `Map`, so the "second run sees the first run's state" invariant the test claims to verify would be silently broken even if the cache-bust worked. The canonical example must be restructured (under Task 5.1.c) so its per-run work lives inside an exported `run(seenSet)` function, and the test calls that function twice within a single module evaluation with a shared seen-set.

Sketch:

````js
import { describe, it, expect } from "vitest";
import { readFileSync, writeFileSync, mkdtempSync } from "node:fs";
import { fileURLToPath, pathToFileURL } from "node:url";
import { dirname, resolve, join } from "node:path";
import { tmpdir } from "node:os";

const __dirname = dirname(fileURLToPath(import.meta.url));
const REPO_ROOT = resolve(__dirname, "../../..");

function extractProgrammaticConsumptionExample() {
  const schemaDoc = readFileSync(
    resolve(REPO_ROOT, "docs/FEED-SCHEMA.md"),
    "utf8",
  );
  // Anchor on the exact section heading so a stray ```js fence elsewhere in the
  // doc cannot satisfy this match. Section heading is "## Programmatic Consumption"
  // at the time of writing; adjust the literal if the heading level changes.
  const SECTION_RE =
    /(^|\n)## Programmatic Consumption\s*\n[\s\S]*?\n```js\n([\s\S]*?)\n```/;
  const sectionMatch = schemaDoc.match(SECTION_RE);
  if (!sectionMatch) {
    throw new Error(
      "Could not find Programmatic Consumption JS example in docs/FEED-SCHEMA.md. " +
        "Has the section heading or fence language changed?",
    );
  }
  return sectionMatch[2];
}

describe("docs/FEED-SCHEMA.md Programmatic Consumption example", () => {
  it("runs against the valid fixture without error and dedupes correctly across two runs", async () => {
    const exampleSource = extractProgrammaticConsumptionExample();
    const fixture = JSON.parse(
      readFileSync(
        resolve(REPO_ROOT, "packages/client/fixtures/all.valid.json"),
        "utf8",
      ),
    );

    // The canonical example in docs/FEED-SCHEMA.md MUST be authored so its
    // per-run logic lives inside an exported `run(seenSet)` function that:
    //   - fetches the feed (via the injected fetch)
    //   - version-gates it
    //   - computes `fresh = items whose uniqueKey is not in seenSet`
    //   - adds each fresh item's uniqueKey into seenSet
    //   - returns `fresh`
    // The test calls `run(seenSet)` twice in a single module evaluation,
    // threading the same seenSet through both calls. See Task 5.1.c for the
    // example's required shape; this test is its enforcement.
    const tmpFile = join(
      mkdtempSync(join(tmpdir(), "aw-docs-example-")),
      "ex.mjs",
    );
    const wrapped = `
      globalThis.__mockFeed = ${JSON.stringify(fixture)};
      globalThis.fetch = async () => ({
        ok: true,
        status: 200,
        json: async () => globalThis.__mockFeed,
      });
      ${exampleSource}
      // The example must export a \`run\` function. If it doesn't, Task 5.1.c
      // has not been satisfied and this test fails at import time.
      export { run };
    `;
    writeFileSync(tmpFile, wrapped);

    const mod = await import(pathToFileURL(tmpFile).href);
    expect(typeof mod.run).toBe("function");

    const seen = new Set();

    const firstFresh = await mod.run(seen);
    expect(Array.isArray(firstFresh)).toBe(true);
    expect(firstFresh.length).toBeGreaterThan(0);
    for (const item of firstFresh) {
      expect(typeof item.uniqueKey).toBe("string");
      expect(item.uniqueKey.length).toBeGreaterThan(0);
    }
    expect(seen.size).toBe(firstFresh.length);

    // Second run against the SAME seen-set and the SAME mocked feed — no new
    // items have arrived, so fresh must be empty. This is the core invariant
    // the test enforces: the example's dedup actually works across runs.
    const secondFresh = await mod.run(seen);
    expect(secondFresh.length).toBe(0);
    expect(seen.size).toBe(firstFresh.length); // unchanged
  });
});
````

**Contract between this test and Task 5.1.c:** the example in `docs/FEED-SCHEMA.md` must export a function with the signature `async function run(seenSet: Set<string>): Promise<Item[]>` that mutates `seenSet` in place with each fresh item's `uniqueKey` and returns the `fresh` array. If the example's shape does not satisfy that contract, this test fails at `expect(typeof mod.run).toBe("function")` and forces Task 5.1.c's restructuring to happen. The test is useless as drift protection if the example isn't shaped correctly; the assertion is what forces the shape.

## Risks and their mitigations

This is the section the review agent should scrutinize most carefully.

### Risk 1: npm publish goes wrong the first time

npm-scoped package publishes fail in subtle ways — wrong access flag, not logged in, two-factor auth interstitial, name conflict with a pre-registered empty scope. If the publish fails partially (version registered but tarball rejected), we cannot re-publish the same version.

**Mitigation:** Task 6.2.5.e runs `npm pack --dry-run` before `npm publish`, inspecting the tarball contents. Task 6.2.5.a sweeps for `<YYYY-MM-DD>` placeholders. Publish only after both are clean. If `npm publish` fails mid-way, bump to `1.0.1` and try again — do not retry `1.0.0`.

### Risk 2: Type generation drifts from hand-written types

`tsc` generates `.d.ts` from JSDoc. If the JSDoc is wrong (e.g., a typedef omits a field that the code actually returns), consumers get misleading autocomplete and TypeScript type errors at their call sites.

**Mitigation:** Task 6.2.5.d (moved pre-publish from the old Task 7.3 — finding M7) runs a TypeScript smoke test in a scratch directory against the locally-built `dist/`, **before** `npm publish` fires. The test imports `AnthropicWatchClient`, `FeedFetchError`, and `type Item`, exercising both the class and error surfaces. If types are wrong, fix JSDoc and regenerate without burning a version number. Task 6.2.5.c additionally greps `dist/index.d.ts` for `Item` / `FeedEnvelope` / `RunReport` to catch the specific failure mode where `src/index.js` fails to re-export typedefs from `./types.js`.

### Risk 3: Fixture divergence between the library's test fixtures and the producer's reference fixtures

`packages/client/fixtures/all.valid.json` is a copy of `docs/fixtures/all.sample.json`. If the producer regenerates its fixtures in a later release and the library doesn't, the library's tests pass against stale shapes.

**Mitigation:** Task 3.5 (`packages/client/test/fixtures.test.js`) asserts byte-identity of the parsed JSON between `packages/client/fixtures/all.valid.json` and `docs/fixtures/all.sample.json`, and the same for the `run-report` pair (finding M8). When the producer regenerates its fixtures in a future release, this test fails and forces the author to re-copy the fixtures into the subpackage in the same PR. Stronger than the originally-planned "schema version matches" check. Long-term, a build step that auto-copies fixtures is possible — deferred to v1.3.1 if this becomes friction.

### Risk 4: Version confusion between the two packages

The repo has two packages with independent versioning. Someone reading the repo might conflate `anthropic-watch@1.3.0` with `@sefaertunc/anthropic-watch-client@1.0.0` and file bugs against the wrong version.

**Mitigation:** CHANGELOG entries in both packages explicitly cross-reference each other. README in each package explicitly notes the package boundary and the sibling version. `docs/WORCLAUDE-INTEGRATION.md` spells out both versions when referring to the pair.

### Risk 5: Consumers that pinned `anthropic-watch` as a dependency (even though they shouldn't have)

The scraper was never published to npm, so this is hypothetical. But someone could theoretically have `"anthropic-watch": "github:sefaertunc/anthropic-watch#v1.2.0"` in a package.json. If they update to `v1.3.0`, they suddenly pull a repo with a `packages/` subdirectory they weren't expecting.

**Mitigation:** Minimal. The scraper was never a public package; anyone pinning it against the git ref has done so at their own risk. The new subdirectory doesn't break their usage — it's strictly additive file-wise.

### Risk 6: Browser consumers hit undefined globals

The library uses `globalThis.fetch` and `AbortController`. Both exist in modern browsers. But if a consumer bundles for legacy browsers (IE11, old iOS), they'll hit runtime errors.

**Mitigation:** Document Node 18+ and "modern browsers" as supported. Do not promise IE or old mobile support. If a bug report arrives, the answer is "use a polyfill or use a supported environment."

### Risk 7: Scope conflicts with a future Anthropic-maintained package

`@anthropic-ai/` is Anthropic's scope. `@sefaertunc/` is yours. There is no conflict today, but if Anthropic ever ships their own `anthropic-watch-client` at `@anthropic-ai/anthropic-watch-client`, consumers might confuse the two.

**Mitigation:** Clear README framing — "unofficial community library." Avoid copying any official Anthropic branding. If Anthropic ever ships a first-party alternative, consider deprecating this one and pointing users there.

### Risk 8: FEED-SCHEMA.md example drifts from library behavior

The inline example in `docs/FEED-SCHEMA.md`'s "Programmatic Consumption" section is the canonical reference for non-JS consumers and pre-adoption evaluators. If the library's real behavior diverges from what the example does (because a bug fix or API refinement only lands in the library), consumers reading the doc are misled.

**Mitigation:** Task 3.6 (`packages/client/test/docs-example.test.js`) programmatically extracts the JavaScript code block from the doc, executes it against a fixture with mocked fetch and mocked state, and asserts the dedup + uniqueKey invariants hold across two simulated runs. If the example drifts from library behavior, CI fails. Review finding Nitpick 3 (modified).

### Risk 9: Root vitest discovers client tests it cannot run

Without intervention, `npm test` at the repo root (using Vitest defaults) would collect and try to run `packages/client/test/*.test.js` — but the root `node_modules/` has not installed the subpackage's devDeps, so any test that imports something only installed there would fail confusingly.

**Mitigation:** Task 1.6 adds `vitest.config.js` at the root with `exclude: ['**/node_modules/**','**/dist/**','packages/**']`. The root test command behaves exactly as it does today; the client package runs via `cd packages/client && npm test`. Two decoupled suites, no cross-package surprise.

### Risk 10: CI never runs `npm ci` inside `packages/client/`

The scraper's CI workflow does `npm ci` at the repo root only. If the subpackage grows a devDep or its lockfile drifts, nobody catches it until the release engineer runs Task 6.2.5 locally.

**Mitigation (accepted residual):** Task 6.2.5.b mandates `cd packages/client && npm ci` before every publish. Task 1.7 expands CI's `pull_request` triggers to include `develop`, so the feature PR at least gets root-level test coverage. Subpackage CI (a matrix job that `cd`s in and runs `npm ci && npm test`) is deferred to v1.3.1 if it becomes friction — the manual pre-publish gate is sufficient for v1.3.0's single coordinated publish moment.
